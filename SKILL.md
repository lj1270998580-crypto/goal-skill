# Goal-Skill 执行手册

## 不可违反的铁律

1. **绝不跳过阶段**：每个阶段所有步骤必须执行完毕，才能进入下一阶段。
2. **绝不走形式**：不能只说"阶段 X 完成"，必须展示实际执行证据。
3. **每个阶段必须展示内容给用户**：不能只在内部分析，每个阶段至少有一个输出要展示给用户看。
4. **状态文件必须更新**：每阶段切换时更新 `.goal-state.md`。
5. **检查点未通过 = 修正，不 = 跳过**：任何检查项失败，必须尝试修正，最多 3 次。3 次失败则上报用户。
6. **阶段开始和结束必须有明确标记**：用固定模板标记阶段边界。
7. **用户确认不 = 省略步骤**：用户说"好的"不代表你不需要执行该阶段的详细步骤。

## 阶段标记模板（必须遵守）

**每个阶段开始和结束必须使用以下格式**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 N 开始】[阶段名称]
  目标：[一句话描述该阶段要做什么]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

阶段执行中展示输出给用户。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 N 结束】[阶段名称]
  检查点：X/X 项通过
  ✅ 进入阶段 N+1
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 阶段 0：启动

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 0 开始】启动
  目标：探测环境、创建状态文件、识别目标类型
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**执行以下步骤**（不可跳过）：

1. `Bash: pwd` → 确认工作目录
2. `Bash: python scripts/goal_state.py init --workspace . --goal "用户目标摘要" --type [software|content|data|document|presentation|general]`
3. `Bash: python --version 2>/dev/null && node --version 2>/dev/null && git --version 2>/dev/null` → 探测环境工具
4. `Read: references/goal-types.md` → 识别目标类型

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 0 结束】启动
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 状态文件已创建 → **验证**：运行 `goal_state.py read --workspace .` 确认文件存在
- [ ] 目标类型已识别 → **验证**：在展示内容中写出识别到的目标类型
- [ ] 环境工具已探测 → **验证**：在展示内容中列出可用工具列表

## 阶段 1：理解目标

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 1 开始】理解目标
  目标：将用户描述转化为清晰、可执行的目标定义
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**执行以下步骤**（不可跳过）：

1. **提取目标**：用 1-2 句话总结用户的核心目标。→ **展示给用户**："我理解你的目标是：..."
2. **识别隐含需求**：列出用户没说但需要的东西。→ **展示给用户**："隐含需求：..."
3. **确认范围**：明确包含什么、不包含什么。→ **展示给用户**："包含：... | 不包含：..."
4. **确认约束**：格式、技术栈、质量标准、截止时间。→ **展示给用户**："约束：..."
5. **提出澄清问题**：提出至少 1 个关键问题。→ **展示给用户**：直接提问
6. **记录状态**：`Bash: python scripts/goal_state.py update-stage --workspace . --stage 1 --status "🟡 In Progress"` → 执行中 → 完成后 `Bash: python scripts/goal_state.py update-stage --workspace . --stage 1 --status "✅ Completed"`

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 1 结束】理解目标
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 核心目标已用 1-2 句话说清 → **验证**：展示给用户的总结是否清晰？用户是否确认？
- [ ] 范围已明确 → **验证**：展示给用户的"包含/不包含"列表是否被用户认可？
- [ ] 已提出至少 1 个澄清问题 → **验证**：用户是否回答了问题？
- [ ] 状态文件已记录 → **验证**：运行 `goal_state.py read --workspace .` 确认阶段 1 状态为 ✅ Completed

## 阶段 2：规划

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 2 开始】规划
  目标：将目标分解为可执行的子任务，生成计划文档
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**执行以下步骤**（不可跳过）：

1. **分解任务**：将目标拆分为 3-10 个可执行子任务，每个子任务有明确输出。→ **展示给用户**：子任务列表
2. **确定依赖顺序**：哪些串行、哪些可以并行。→ **展示给用户**：依赖关系（如"任务 A 必须在 B 之前"）
3. **识别所需技能**：是否需要加载其他 skill。→ **展示给用户**：技能列表
4. **生成计划文件**：`Write: goal-plan.md`（子任务列表、技能映射、依赖关系）→ **展示给用户**：计划文件路径
5. **用户确认**：向用户呈现计划摘要（3-5 句话），请求确认。→ **展示给用户**：计划摘要 + 确认请求
6. **记录状态**：`Bash: python scripts/goal_state.py update-stage --workspace . --stage 2 --status "🟡 In Progress"` → `Bash: python scripts/goal_state.py update-stage --workspace . --stage 2 --status "✅ Completed"`

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 2 结束】规划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 子任务数量在 3-10 之间，每个有明确输出 → **验证**：展示给用户的子任务列表是否清晰？每个任务是否有一行描述输出？
- [ ] 计划覆盖了用户目标的全部范围 → **验证**：对照阶段 1 的"包含"列表，逐一确认每个包含项都有对应子任务
- [ ] 计划文件已创建 → **验证**：`goal-plan.md` 是否存在？`Bash: ls goal-plan.md`
- [ ] 用户已确认或已明确说明不需要确认 → **验证**：用户是否回复了"确认"或"同意"？
- [ ] 状态文件已记录 → **验证**：运行 `goal_state.py read --workspace .` 确认阶段 2 状态为 ✅ Completed

## 阶段 3：设计

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 3 开始】设计
  目标：创建架构或结构设计方案
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**设计阶段不可跳过**，除非目标纯粹是文本生成（如"写 100 字总结"）。

**判断是否可以简短处理**：
- 纯文本生成（无结构、无架构）→ 设计阶段可以简化为 1-2 句话的结构说明，写入 `goal-design.md`
- 任何涉及软件、结构、布局、数据流的目标 → 必须完整设计文档

**执行以下步骤**（不可跳过）：

1. **加载设计标准**：`Read: references/development-standards.md`（软件项目读"架构"章节，内容项目读"UI/UX"章节）→ **展示给用户**："加载设计标准：UI/UX 规范 / 架构标准"
2. **创建设计文档**：`Write: goal-design.md`
   - 软件：架构、API 契约、数据模型、组件边界
   - 内容：大纲、章节结构、叙事流
   - 数据：分析流程、可视化方案
   - → **展示给用户**：设计摘要（3-5 个关键设计决策）
3. **设计自检**：对照阶段 1 的目标验证设计是否满足。→ **展示给用户**：自检结果（"设计覆盖了 X 个需求点"）
4. **记录状态**：`Bash: python scripts/goal_state.py update-stage --workspace . --stage 3 --status "🟡 In Progress"` → `Bash: python scripts/goal_state.py update-stage --workspace . --stage 3 --status "✅ Completed"`

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 3 结束】设计
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 设计文档已创建 → **验证**：`Bash: ls goal-design.md`
- [ ] 设计满足需求 → **验证**：对照阶段 1 的"包含"列表，逐一确认每个需求都有对应设计
- [ ] 设计遵循标准 → **验证**：展示给用户的设计是否提到了标准遵循（如"使用 8px 网格系统"）
- [ ] 状态文件已记录 → **验证**：运行 `goal_state.py read --workspace .` 确认阶段 3 状态为 ✅ Completed

## 阶段 4：执行

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 4 开始】执行
  目标：按计划执行子任务，产生实际输出
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**执行以下步骤**（不可跳过）：

1. **加载领域技能**：如果计划阶段识别了需要加载的技能，现在加载。→ **展示给用户**："加载技能：xxx"
2. **按序执行子任务**：每个子任务完成后：
   - 展示该子任务的输出（代码片段、文件路径、内容摘要）
   - 每完成 1-2 个任务向用户展示进度
3. **记录决策**：执行过程中关键决策写入执行日志。→ **展示给用户**：至少 1 个关键决策
4. **保存中间产物**：重要文件及时保存。→ **展示给用户**：保存的文件列表
5. **记录状态**：`Bash: python scripts/goal_state.py update-stage --workspace . --stage 4 --status "🟡 In Progress"` → `Bash: python scripts/goal_state.py update-stage --workspace . --stage 4 --status "✅ Completed"`

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 4 结束】执行
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 所有子任务已执行 → **验证**：对照 `goal-plan.md` 的子任务列表，逐一确认每个都有输出
- [ ] 输出符合预期 → **验证**：展示给用户的每个子任务输出是否可验证？（如代码文件存在、内容正确）
- [ ] 是否按设计执行 → **验证**：如有设计阶段，对照 `goal-design.md` 验证实现是否一致
- [ ] 状态文件已记录 → **验证**：运行 `goal_state.py read --workspace .` 确认阶段 4 状态为 ✅ Completed

## 阶段 5：验证

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 5 开始】验证
  目标：验证结果质量和完整性
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**执行以下步骤**（不可跳过）：

1. **加载检查清单**：`Read: references/quality-checklist.md` → 通用检查清单 + 领域检查清单
2. **逐项验证**：对每个检查项，必须说明验证方法：
   - 能自动验证的：实际运行（如"代码可运行" → 实际运行测试命令）
   - 需人工判断的：说明判断依据（如"内容逻辑连贯" → 说明哪里连贯/不连贯）
3. **记录验证结果**：对每个检查项记录 ✅/❌ 和证据
4. **修复缺陷**：发现 ❌ 项 → 修复 → 重新验证（最多 3 轮）→ **展示给用户**：每次修复后的重新验证结果
5. **生成验证报告**：`Write: goal-verification-report.md`（检查项列表、通过/失败项、修复情况）→ **展示给用户**：验证报告摘要（X/Y 项通过）
6. **记录状态**：`Bash: python scripts/goal_state.py update-stage --workspace . --stage 5 --status "🟡 In Progress"` → `Bash: python scripts/goal_state.py update-stage --workspace . --stage 5 --status "✅ Completed"`

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 5 结束】验证
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 检查清单已逐项执行 → **验证**：展示给用户的验证结果是否包含每项的证据？
- [ ] 所有检查项有证据 → **验证**：每个 ✅ 是否有验证动作说明？（如"运行 `python test.py` → 输出 OK"）
- [ ] 结果覆盖全部范围 → **验证**：对照阶段 1 的"包含"列表，确认所有包含项都有对应验证
- [ ] 验证报告已生成 → **验证**：`Bash: ls goal-verification-report.md`
- [ ] 状态文件已记录 → **验证**：运行 `goal_state.py read --workspace .` 确认阶段 5 状态为 ✅ Completed

## 阶段 6：交付

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 6 开始】交付
  目标：将最终成果呈现给用户
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**执行以下步骤**（不可跳过）：

1. **组装交付物**：列出所有输出文件。→ **展示给用户**：交付物清单（文件路径列表）

4. **正式交付**：向用户呈现文件路径和摘要。→ **展示给用户**："交付完成：以下文件已生成..."
5. **记录状态**：`Bash: python scripts/goal_state.py update-stage --workspace . --stage 6 --status "🟡 In Progress"` → `Bash: python scripts/goal_state.py update-stage --workspace . --stage 6 --status "✅ Completed"`

  【阶段 6 结束】交付
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**检查点（逐项验证）**：
- [ ] 交付物清单完整 → **验证**：展示给用户的文件列表是否包含所有输出？
- [ ] 使用说明至少 3 步 → **验证**：展示给用户的步骤数 >= 3？
- [ ] 已知限制已说明 → **验证**：展示给用户的限制列表是否存在（即使为空也要说明"无已知限制"）
- [ ] 状态文件已记录 → **验证**：运行 `goal_state.py read --workspace .` 确认阶段 6 状态为 ✅ Completed

## 阶段 7：反思（用户要求时）

**━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【阶段 7 开始】反思
  目标：评估交付质量，发现改进点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━**

**只在用户明确说"优化"、"改进"、"修复问题"时执行。**

**执行以下步骤**（不可跳过）：

1. `Read: references/evolution-loop.md`
2. 按 6 维度评估：功能、Bug、体验、架构、性能、安全。→ **展示给用户**：每个维度的评分和发现
3. 计算总分（1-5 分/维度，总分 30）。→ **展示给用户**：总分和评级
4. 生成改进清单，按优先级排序。→ **展示给用户**：改进清单（高/中/低优先级）
5. 决策：
   - 评分 >= 25/30 且改进点 <= 2 → **展示给用户**："质量已达标，无需进一步优化"
   - 改进点 > 0 且当前轮次 < 6 → **展示给用户**："进入第 X 轮优化..."
   - 轮次 == 6 → **展示给用户**："已达到最大进化轮次，请用户决策"
6. **记录状态**：`Bash: python scripts/goal_state.py retrospective --workspace . --round N --scores "func=X,bug=X,ux=X,code=X,perf=X,sec=X" --improvements "N"`

## 上报规则（必须遵守）

以下情况必须停止执行并上报用户：

1. 缺少关键信息（只有用户能提供）
2. 环境不满足（缺少必要工具/权限）
3. 同一问题 3 次修正失败
4. 目标矛盾或不可行
5. 用户明确说"停"或"算了"

**上报格式**：
```
🛑 执行暂停 — 需要用户协助
【当前阶段】：X
【问题】：具体描述
【已尝试】：1)... 2)... 3)...
【选项】：A) ... B) ...
```
