# Evolution Loop — 反思与进化循环

加载此文件在 **阶段 6 交付完成后**，用于自动反思和进入下一轮优化。

---

## 1. 核心概念

目标完成不是终点。交付后，AI 必须主动反思：
- 是否还有未发现的 bug？
- 功能是否完整？
- 用户体验是否足够好？
- 代码质量是否达标？
- 架构是否合理？

**进化循环**：交付 → 反思 → 发现改进点 → 回到阶段 3/4 修复 → 重新交付 → 再次反思 → ...（最多 6 轮）

---

## 2. 进化轮次模型

```
第 1 轮（初始目标）：
  用户目标 → 理解 → 规划 → 设计 → 执行 → 验证 → 交付
  → 反思 → 生成改进清单

第 2 轮（第 1 次进化）：
  改进清单 → 理解变更 → 重新规划 → 设计修复 → 执行修复 → 验证 → 交付
  → 反思 → 生成改进清单

第 3 轮（第 2 次进化）：
  改进清单 → ... → 反思 → 生成改进清单

...

第 6 轮（第 5 次进化）：
  ... → 反思 → 如果没有改进点 → 结束
  → 如果还有改进点 → 上报用户："已完成 6 轮进化，仍发现改进点：... 是否继续？"
```

**轮次规则**：
- 第 1 轮是用户原始目标
- 第 2-6 轮是基于反思的自动优化
- 每轮结束后都进行反思
- 如果反思发现无改进点，提前结束
- 如果 6 轮后仍有改进点，上报用户决策

---

## 3. 阶段 7：反思（Retrospective）

**触发时机**：阶段 6 交付完成后，用户确认收到交付物后

**目的**：系统性地审查交付结果，发现改进点

**过程**：

### 3.1 反思维度检查清单

对每个维度进行评估（满分 5 分，1=差，5=优秀）：

#### 维度 A：功能完整性
- [ ] 所有用户要求的功能都已实现？
- [ ] 功能是否按预期工作？
- [ ] 是否有遗漏的边界情况？
- [ ] 是否有用户未明说但隐含需要的功能？
- [ ] 功能之间的交互是否正确？

**评分**：____ / 5
**改进点**：

#### 维度 B：Bug 与缺陷
- [ ] 是否有运行时错误？
- [ ] 是否有功能不生效的情况？
- [ ] 是否有明显的逻辑错误？
- [ ] 是否有数据丢失或损坏的风险？
- [ ] 是否有未处理的异常或错误？

**评分**：____ / 5
**改进点**：

#### 维度 C：用户体验（UI/UX）
- [ ] 界面是否清晰、直观？
- [ ] 交互是否流畅、无卡顿？
- [ ] 是否有良好的视觉反馈？
- [ ] 是否有适当的错误提示和引导？
- [ ] 响应式/适配性是否良好？（如适用）
- [ ] Logo/品牌标识是否专业？（如适用）

**评分**：____ / 5
**改进点**：

#### 维度 D：架构与代码质量
- [ ] 代码结构是否清晰、模块化？
- [ ] 是否有冗余代码或重复逻辑？
- [ ] 命名是否清晰、一致？
- [ ] 是否有适当的注释和文档？
- [ ] 依赖是否合理、可管理？
- [ ] 是否遵循了设计模式或架构标准？

**评分**：____ / 5
**改进点**：

#### 维度 E：性能
- [ ] 启动/加载速度是否可接受？
- [ ] 内存/CPU 使用是否合理？
- [ ] 是否有明显的性能瓶颈？
- [ ] 大数据量处理是否高效？（如适用）
- [ ] 是否有缓存/优化策略？

**评分**：____ / 5
**改进点**：

#### 维度 F：安全与稳定性
- [ ] 是否有 SQL 注入、XSS 等安全风险？
- [ ] 是否有硬编码的敏感信息？
- [ ] 输入验证是否充分？
- [ ] 是否有适当的权限控制？
- [ ] 错误处理是否健壮？

**评分**：____ / 5
**改进点**：

### 3.2 反思评分汇总

```
反思报告 — 第 N 轮

| 维度 | 评分 | 改进点数量 | 优先级 |
|------|------|-----------|--------|
| 功能完整性 | X/5 | N | 高/中/低 |
| Bug 与缺陷 | X/5 | N | 高/中/低 |
| 用户体验 | X/5 | N | 高/中/低 |
| 架构与代码质量 | X/5 | N | 高/中/低 |
| 性能 | X/5 | N | 高/中/低 |
| 安全与稳定性 | X/5 | N | 高/中/低 |
| **总分** | XX/30 | **N 个改进点** | |

**改进清单**（按优先级排序）：
1. [高] ...
2. [高] ...
3. [中] ...
4. [中] ...
5. [低] ...
```

### 3.3 进化决策

**如果评分 >= 25/30 且改进点 <= 2**：
→ 质量已达标，无需进一步进化。向用户展示反思报告，结束。

**如果改进点 > 0 且当前轮次 < 6**：
→ 进入下一轮进化。向用户展示：
```
✅ 第 N 轮完成，已交付。

📊 反思报告：
  - 总体评分：XX/30
  - 发现 N 个改进点（X 高优先级、X 中优先级、X 低优先级）

🔧 主要改进方向：
  1. [高] ...
  2. [高] ...

🔄 建议进入第 N+1 轮优化，重点修复上述问题。

请选择：
  ✅ 自动优化（进入第 N+1 轮）
  🎨 只优化高优先级项
  ⏹ 停止，当前版本已满足需求
```

**如果当前轮次 == 6**：
→ 这是最后一轮。向用户展示：
```
📊 第 6 轮反思报告：
  - 总体评分：XX/30
  - 仍发现 N 个改进点

⚠️ 已达到最大进化轮次（6/6）。

后续建议：
  - 高优先级改进点：...
  - 中优先级改进点：...
  - 低优先级改进点：...

请选择：
  🔄 继续优化（手动开启新一轮）
  ⏹ 结束，当前版本为最终版
```

## 4. 进化执行流程

### 4.1 进化轮次不是重做一次全部

第 N+1 轮的执行策略：

```
改进清单分析
  │
  ├─ 只涉及设计层面？→ 回到阶段 3（Design）修改 → 阶段 4 → 5 → 6
  ├─ 只涉及代码修复？→ 回到阶段 4（Execute）修复 → 阶段 5 → 6
  ├─ 涉及需求变更？→ 回到阶段 2（Plan）调整 → 阶段 3 → 4 → 5 → 6
  └─ 涉及目标理解？→ 回到阶段 1（Understand）重新理解
```

**最小回溯原则**：只回到最早受影响的阶段，不要每次都从阶段 1 开始。

### 4.2 进化计划生成

将改进清单转化为可执行的子任务：

```markdown
# Evolution Plan — 第 N+1 轮

**目标**：修复第 N 轮反思发现的问题

**改进清单**：
1. [高] 修复登录功能在空密码时崩溃（阶段 4）
2. [高] 添加错误处理机制（阶段 4）
3. [中] 优化按钮样式（阶段 3/4）
4. [中] 添加 README 文档（阶段 4）

**执行策略**：
- 回到阶段 4（Execute），修改代码
- 不涉及阶段 1-3 的变更
- 完成后执行阶段 5（Verify）和阶段 6（Deliver）
```

## 5. 反思触发条件

反思不是每轮都强制，而是根据以下策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **自动反思**（默认） | 每轮交付后自动执行反思 | 软件开发、复杂应用 |
| **用户触发** | 用户说"优化一下"或"修复 bug" | 用户主动要求 |
| **问题触发** | 执行中发现问题，立即反思 | 发现严重问题 |
| **时间触发** | 达到预设时间后自动反思 | 长时间运行任务 |

**推荐策略**：软件开发默认使用"自动反思"，其他类型使用"用户触发"。

---

## 6. 反思深度控制

根据目标复杂度，控制反思深度：

| 复杂度 | 反思维度 | 检查项数量 | 时间投入 |
|--------|----------|-----------|----------|
| **轻量** | 只检查功能和 Bug | 6-10 项 | 少 |
| **标准** | 功能 + Bug + 代码质量 | 15-20 项 | 中等 |
| **深度** | 全部 6 个维度 | 30+ 项 | 多 |

---

## 7. 进化结束条件

以下任一条件满足时，结束进化循环：

1. **质量达标**：评分 >= 25/30 且改进点 <= 2
2. **轮次上限**：完成 6 轮进化
3. **用户停止**：用户明确说"停止"或"足够了"
4. **无改进点**：反思发现 0 个改进点
5. **收益递减**：连续 2 轮反思发现的主要改进点相同（说明进入优化瓶颈）

## 9. 快速检查表（反射检查）

每轮反思时，先快速回答这 6 个问题：

1. **用户会满意吗？** — 如果用户现在看到结果，会说什么？
2. **我能信任这个代码吗？** — 如果这是一个生产系统，我敢部署吗？
3. **还有明显的 bug 吗？** — 用 3 分钟测试，能发现什么？
4. **代码是我想维护的吗？** — 如果 3 个月后回来看，我能理解吗？
5. **性能会让用户抱怨吗？** — 加载超过 3 秒？操作卡顿？
6. **安全上有没有明显漏洞？** — 输入没验证？明文存密码？

如果 6 个问题都回答"是/没问题" → 进入结束检查。
如果任何一个是"否/有问题" → 深入检查对应维度。
