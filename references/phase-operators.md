# Phase Operators — 阶段操作手册

加载此文件在 **每个阶段执行时**，作为该阶段的"操作手册"，告诉 AI 具体"怎么做"。

---

## 1. 检查点执行方式（How to Run a Checkpoint）

检查点不是"读一遍清单"，而是**逐项验证、记录结果、做出判断**。

### 1.1 检查点执行流程

```
开始检查点
  │
  ▼
读取当前阶段的所有检查项
  │
  ▼
对每一项：
  ├─ 能自动验证？→ 执行验证操作 → 记录结果 ✅/❌
  └─ 需要人工判断？→ 基于证据做出判断 → 记录结果 ✅/❌
  │
  ▼
统计：全部通过？→ 阶段通过，进入下一阶段
  │
  └─ 有未通过？→ 分类处理（见 1.2）
```

### 1.2 自动验证 vs 人工判断

| 检查项类型 | 示例 | 验证方式 | 证据来源 |
|-----------|------|---------|---------|
| **自动验证** | "代码可以运行" | 执行编译/运行命令 | 命令输出 |
| **自动验证** | "文件存在" | 检查文件系统 | `ls` / `Read` |
| **自动验证** | "格式正确" | 尝试解析/渲染 | 解析器结果 |
| **人工判断** | "设计满足需求" | 对比设计文档与需求 | 文档对比 |
| **人工判断** | "内容逻辑连贯" | 阅读内容并判断 | 内容分析 |
| **人工判断** | "质量达标" | 基于标准评估 | 经验判断 |

**规则**：
- 能自动验证的，**必须**自动验证，不能只凭"感觉"
- 自动验证失败时，记录具体错误信息
- 人工判断时，**必须说明判断依据**（不能只说"我觉得可以"）

### 1.3 检查点结果记录格式

每次检查点执行后，在状态文件中记录：

```markdown
## Checkpoint — Stage X — [timestamp]

| 检查项 | 结果 | 验证方式 | 证据/备注 |
|--------|------|---------|-----------|
| 项 1 | ✅ | 自动：运行测试 | 测试全部通过 |
| 项 2 | ✅ | 人工判断 | 对比需求文档，覆盖全部 5 个需求 |
| 项 3 | ❌ | 自动：格式检查 | 缺少目录页 |
| 项 4 | ✅ | 自动：文件检查 | 所有文件已创建 |
| 项 5 | — | 未执行 | 因项 3 失败，暂停 |

**结论**：❌ 未通过（1/4 项失败）
**处理**：修复项 3（缺少目录），重新验证
**重试次数**：1/3
```

## 2. 各阶段具体操作指令

### 2.1 Stage 1: Understand — 具体做什么

**步骤 1.1：读取用户输入**
- 提取用户原始请求的全部内容
- 识别是否有附件、参考文件、URL

**步骤 1.2：目标解析**
- 用一句话总结用户的核心目标
- 识别目标类型（读取 `references/goal-types.md` 决策树）
- 识别范围边界（包含/排除）

**步骤 1.3：约束提取**
- 格式要求（`.docx`、`.pptx`、代码语言等）
- 质量标准（"生产级"、"原型"、"草稿"）
- 技术栈（如果用户指定了）
- 时间/资源限制（如果用户提到了）

**步骤 1.4：信息缺口检测**
- 列出为了执行目标，目前还缺失的信息
- 按重要性排序：必须有 / 最好有 / 可以有
- 对"必须有"的信息，准备澄清问题

**步骤 1.5：执行检查点**
- 按照 Stage 1 Checkpoint Gate 逐项验证
- 如果信息不足，先向用户提问，再验证

**步骤 1.6：用户确认（标准/深度路径）**
- 向用户呈现目标理解摘要
- 请求确认或修正
- 记录用户反馈到状态文件

---

### 2.2 Stage 2: Plan — 具体做什么

**步骤 2.1：选择执行路径**
- 读取复杂度评估结果（来自启动协议）
- 轻量路径：跳过设计阶段，简化检查点
- 标准路径：完整六阶段
- 深度路径：增加预研和评审

**步骤 2.2：技能识别**
- 根据目标类型，确定需要加载的领域技能
- 读取 `references/goal-types.md` 技能映射表
- 记录每个技能的加载阶段

**步骤 2.3：任务分解**
- 将目标分解为 3-10 个可执行子任务
- 每个子任务有：名称、输出、依赖、估计时间
- 识别哪些可以并行执行

**步骤 2.4：依赖排序**
- 绘制子任务依赖图
- 确定执行顺序（串行/并行）
- 识别关键路径

**步骤 2.5：计划文档化**
- 将计划写入 `{workspace}/goal-plan.md`
- 包含：子任务列表、依赖图、技能映射、时间表（如果适用）

**步骤 2.6：执行检查点**
- 按照 Stage 2 Checkpoint Gate 逐项验证计划质量

**步骤 2.7：用户确认（标准/深度路径）**
- 向用户呈现计划摘要
- 请求确认或调整
- 用户确认后，锁定计划并进入下一阶段

---

### 2.3 Stage 3: Design — 具体做什么

**步骤 3.1：设计准备**
- 读取计划文档 `goal-plan.md`
- 确认设计范围（只对需要设计的部分进行设计）

**步骤 3.2：创建设计文档**
- 根据目标类型创建设计文档：
  - 软件：`goal-design.md` 包含架构图、API 契约、数据模型
  - 内容：`goal-design.md` 包含大纲、章节结构、叙事流
  - 数据：分析计划、可视化方案
  - 演示：幻灯片结构、视觉方向

**步骤 3.3：设计评审**
- 自我评审：设计是否满足需求？是否有明显缺陷？
- 记录设计决策（为什么这样设计，而不是其他方案）

**步骤 3.4：执行检查点**
- 按照 Stage 3 Checkpoint Gate 逐项验证

**步骤 3.5：用户确认（设计评审）**
- 向用户呈现设计摘要
- 请求确认或修改
- 用户确认后，锁定设计

---

### 2.4 Stage 4: Execute — 具体做什么

**步骤 4.1：加载领域技能**
- 根据计划加载需要的技能
- 读取技能文件，理解其工作流程

**步骤 4.2：按序执行子任务**
- 按照依赖顺序执行每个子任务
- 每完成一个子任务，更新状态文件
- 记录执行日志

**步骤 4.3：执行中监控**
- 每完成 1-2 个子任务，向用户展示进度（长执行时）
- 检查是否有用户中途变更（读取用户最新输入）
- 执行元认知检查（我是否偏离了目标？）

**步骤 4.4：处理失败子任务**
- 分析失败原因
- 尝试不同方案（重试 1-3 次）
- 3 次失败 → 上报（不是等待，而是立即上报）

**步骤 4.5：执行检查点**
- 所有子任务完成后，执行 Stage 4 Checkpoint Gate
- 验证所有输出是否符合预期

---

### 2.5 Stage 5: Verify — 具体做什么

**步骤 5.1：通用检查**
- 读取 `references/quality-checklist.md` 通用检查清单
- 逐项验证（能自动的自动验证，需人工的判断）
- 记录每一项的结果

**步骤 5.2：领域检查**
- 根据目标类型，读取对应的领域检查清单
- 逐项验证
- 记录缺陷列表

**步骤 5.3：缺陷修复**
- 对发现的缺陷：
  - 致命缺陷 → 修复后重新验证
  - 严重缺陷 → 修复后重新验证
  - 轻微缺陷 → 修复（如果时间允许）或记录
- 记录修复情况

**步骤 5.4：验证报告**
- 将验证结果写入 `{workspace}/goal-verification-report.md`
- 包含：检查项清单、通过/失败项、缺陷列表、修复情况、最终结论

**步骤 5.5：执行检查点**
- 按照 Stage 5 Checkpoint Gate 验证
- 如果未通过，允许最多 3 轮修复-验证循环
- 3 轮后仍有缺陷 → 上报用户

---

### 2.6 Stage 6: Deliver — 具体做什么

**步骤 6.1：组装交付物**
- 收集所有输出文件
- 确认文件完整性和正确性
- 如有必要，打包或转换格式

**步骤 6.2：编写交付说明**
- 使用说明（如何运行/使用交付物）
- 依赖项清单（如果需要安装）
- 已知限制（如实告知）
- 后续建议（可选改进方向）

**步骤 6.3：执行最终检查点**
- 按照 Stage 6 Checkpoint Gate 逐项验证

**步骤 6.4：正式交付**
- 向用户呈现交付物
- 提供文件路径和简要说明
- 交付说明文档

**步骤 6.5：用户满意度确认**
- 询问用户是否满意
- 如果用户不满意，记录反馈，触发修正流程（回到相关阶段）
- 如果用户满意，标记完成，归档状态文件

## 3. 重试的具体操作

### 3.1 重试策略矩阵

| 失败类型 | 第 1 次重试 | 第 2 次重试 | 第 3 次重试 |
|---------|------------|------------|------------|
| 工具调用失败 | 直接重试 | 等待 2 秒后重试 | 换用备用工具 |
| 参数错误 | 修正参数 | 简化参数 | 换用不同方法 |
| 环境限制 | 找 workaround | 降級方案 | 上报用户 |
| 逻辑错误 | 修正逻辑 | 重写相关部分 | 上报用户 |
| 外部依赖失败 | 等待 1s 重试 | 等待 2s 重试 | 等待 4s 重试 |

### 3.2 重试记录格式

```markdown
## Retry Log — [Subtask/Checkpoint] — [timestamp]

**原始失败**：[错误描述]

**重试 1**：[方案] → [结果]
**重试 2**：[方案] → [结果]
**重试 3**：[方案] → [结果]

**最终决策**：[通过 / 上报]
**上报内容**：[如果上报，记录上报内容]
```

## 4. 用户中途交互处理

### 4.1 执行中用户说话了

如果在执行过程中用户发送了新消息，AI 必须：

1. **暂停当前操作**（不中断已进行的工具调用，但暂停下一步）
2. **判断消息类型**：
   - 问题/询问 → 回答后继续执行
   - 变更请求 → 触发变更处理流程
   - 取消 → 触发取消处理流程
   - 催促 → 展示进度，继续执行
   - 反馈 → 记录反馈，继续执行
3. **更新状态文件**记录用户交互

### 4.2 变更处理（详细）

**小变更**（不影响架构/计划）：
- 在当前阶段直接处理
- 例如：修改措辞、调整颜色、增加一个小功能
- 处理完后继续执行

**中变更**（影响部分子任务）：
- 评估影响：哪些子任务需要重做？
- 回到最早受影响的阶段
- 修改计划/设计
- 重新执行受影响的部分
- 不需要回到阶段 1

**大变更**（改变目标方向）：
- 相当于新目标
- 询问用户是否确认（"这会重置当前进度，是否继续？"）
- 如果确认：重置状态文件，从阶段 1 重新开始
- 如果不确认：继续原目标

### 4.3 取消处理

如果用户说"算了"、"不做了"、"取消"：

1. 立即停止执行
2. 向用户确认："确认取消吗？已完成的 X 阶段将被保留。"
3. 如果用户确认取消：
   - 更新状态文件：标记为"已取消"
   - 记录已完成的部分
   - 告知用户已保存的中间产物
4. 如果用户不确认：继续执行

## 5. 时间/资源预算管理

### 5.1 预算声明

如果用户明确提到时间或资源限制（如"尽快"、"简单做"、"不要太多 token"）：

- 在阶段 1 记录预算约束
- 在阶段 2 选择更轻量的路径
- 在执行中定期检查是否超出预算
- 如果可能超出预算，提前告知用户并请求决策

### 5.2 预算检查点

- 阶段 2 结束时：确认计划符合预算
- 阶段 4 中期：检查执行进度是否按预期
- 阶段 5 开始时：确认剩余资源足以完成验证

## 6. 元认知检查操作

### 6.1 何时执行

- 每个阶段完成时
- 每完成 2-3 个子任务时（长执行中）
- 遇到意外情况时
- 用户长时间无响应后恢复时

### 6.2 检查问题清单

```
【元认知检查】[timestamp]
- [ ] 我当前做的事情是否仍然服务于用户的原始目标？
  → 如果不确定，重读阶段 1 的目标定义
- [ ] 我是否偏离了计划？如果是，为什么？是否合理？
  → 如果偏离不合理，回到计划
- [ ] 我是否在做超出当前阶段的事情？
  → 如果是，停止，回到当前阶段
- [ ] 上下文是否被污染？（是否混入了无关信息）
  → 如果是，重新聚焦
- [ ] 我是否需要重新确认用户的意图？
  → 如果是，向用户确认
- [ ] 当前进度是否仍然可控？
  → 如果不可控，考虑上报或简化
```

### 6.3 检查结果处理

如果任何一项答案为"否"或"需要"：
- 暂停执行
- 分析问题
- 采取措施：修正方向 / 重新确认 / 简化范围 / 上报
- 记录到执行日志
